# Valkyrja Data Cache Files

## Core Concept

Data cache files are pre-generated source files containing fully resolved,
statically defined route, listener, command, and container binding data. They
allow the framework to skip the bootstrap process entirely on cold starts —
instead of loading providers, scanning annotations, building dispatchers, and
resolving handlers, the application loads pre-built data classes directly.

The build tool generates exactly **four data classes** for the entire
application — one per concern — each aggregating everything across all providers
of that type:

| Class                | Contains                                                             |
|----------------------|----------------------------------------------------------------------|
| `AppContainerData`   | All bindings from every `ServiceProvider` across all components      |
| `AppEventData`       | All listeners from every `ListenerProvider` across all components    |
| `AppHttpRoutingData` | All HTTP routes from every `HttpRouteProvider` across all components |
| `AppCliRoutingData`  | All CLI routes from every `CliRouteProvider` across all components   |

This means the framework loads exactly four data classes at boot. No merging, no
iteration, no construction — the structures are immediately ready to use.

Cache data files are **required for production CGI and lambda deployments**
across all languages. They are **optional for worker mode deployments** and *
*not needed in development**.

---

## The Problem They Solve

Without cache data files, every CGI request or lambda cold start pays the full
bootstrap cost:

- Traverse the full provider tree (AppProvider → ComponentProviders →
  SubProviders)
- Register all container bindings
- Scan annotations / walk route registrations
- Build route dispatcher index
- Resolve handler closures
- Register all event listeners and CLI commands

At framework scale — Container, Http, Routing, Dispatcher, Event, CLI, Queue,
Cache, Session, Auth and more — the binding count is not small and the provider
chain traversal has real cost. Cache data files reduce cold start cost to
near-zero by pre-computing all of the above into static data classes that are
simply loaded.

**Worker mode** pays this cost once at startup and amortizes it across all
requests — cache is an optimization.

**CGI / lambda mode** pays this cost on every single request or cold start —
cache is a hard production requirement.

---

## The valkyrja-build Tool

Cache data files are generated by `valkyrja-build` — a standalone tool that
ships with the framework. The framework itself has zero AST or build dependency.
All source extraction logic lives in the build tool.

The build tool:

1. Reads the application config class to discover top-level component providers
2. Walks the static provider tree via AST to discover all sub-providers
3. Categorizes each leaf provider (container / http route / cli route / event)
4. Extracts all bindings, routes, listeners, and commands across every provider
   of each type
5. Resolves all type references to fully-qualified names
6. For routes: constructs plain ValkyrjaRoute objects, runs ProcessorContract::
   route() to compile regexes, pre-builds all dispatcher indexes
7. Writes exactly **four** aggregated data classes — one per concern for the
   entire application
8. Generated files are compiled/included in the final deployment artifact

The build tool is a **source code generator** — it writes strings that are valid
source code. It does not need to instantiate, extend, or have access to
application-defined custom classes at generation time. Class names from AST are
written as text into the generated files. The compiler/runtime resolves them
when the generated files are compiled/loaded alongside the application.

---

## The Provider Tree

The build tool reads the application config class and discovers everything else
by walking the static provider tree:

```
AppProvider (read from AppConfig.providers)
  └── HttpComponentProvider
        ├── HttpContainerProvider       ← container bindings
        ├── HttpMiddlewareProvider      ← container bindings
        ├── HttpEventProvider           ← listeners
        └── HttpRouteProvider           ← routes
  └── ContainerComponentProvider
        ├── ContainerBindingsProvider   ← container bindings
        └── ContainerAliasProvider      ← container bindings
  └── App\Providers\AppContainerProvider    ← application bindings
  └── App\Http\Providers\UserRouteProvider  ← application routes
```

Provider list methods must return **simple list literals with no conditional
logic**. This is a hard framework convention — the build tool fails with a clear
error if it encounters dynamic returns.

---

## Application Config — Build Tool Entry Point

No separate `valkyrja.yaml` file is needed. The application config class is
already the authoritative list of component providers — it is the build tool's
entry point.

```php
// PHP — build tool reads providers list from AppConfig via AST
$app = Application::create(
    new AppConfig(
        providers: [
            HttpComponentProvider::class,
            ContainerComponentProvider::class,
            EventComponentProvider::class,
            CliComponentProvider::class,
            App\Providers\AppProvider::class,
        ]
    )
);
```

Provider class references must use `::class` / `.class` / class objects
directly — never string constants from a constants class. The build tool cannot
follow constant references without executing code.

```
✅ HttpComponentProvider::class           — readable from AST
❌ HttpConstants::HTTP_COMPONENT_PROVIDER  — constant reference, not resolvable
```

The component provider constants class is not part of the framework for this
reason. Binding key constants files (for container bindings) are unaffected —
they are never used in provider lists.

Pre-generated framework and third-party cache files are discovered by the build
tool from conventional locations in the vendor directory — no explicit listing
needed.

## The Four Generated Data Classes

The build tool aggregates everything across all providers of each type into
exactly four classes per layer (framework, third-party packages, application):

**AppContainerData** — all bindings from every `ServiceProvider` across all
components and the application, fully keyed and ready for the container to load
directly.

**AppEventData** — all listeners from every `ListenerProvider` across all
components and the application, keyed by event type.

**AppHttpRoutingData** — all HTTP routes from every `HttpRouteProvider` across
all components and the application. Uses the `HttpRoutingData` structure with
all four indexes pre-built:

```php
readonly class HttpRoutingData
{
    public function __construct(
        public array $routes      = [], // all routes keyed by name/identifier
        public array $paths       = [], // static paths: method → path → route key
        public array $dynamicPaths = [], // dynamic paths: method → path → route key
        public array $regexes     = [], // compiled regexes: method → regex → route key
    ) {}
}
```

**AppCliRoutingData** — all CLI routes from every `CliRouteProvider` across all
components and the application, equivalent structure to HTTP routing data.

---

## Cache Load Order

The application ships two sets of four data classes — framework pre-generated
and application generated. Framework data loads first, application data merges
on top. Application wins on any key conflict:

```
1. Framework pre-generated data classes    ← base bindings, routes, listeners
        ↓ merge — application overrides framework
2. Application generated data classes      ← application bindings, routes, listeners
        ↓
3. Final merged state loaded into runtime
```

This is **one merge of two pre-built structures** — not N merges of N provider
outputs. The build tool does the aggregation at generation time so the framework
pays minimal cost at boot.

Third-party packages insert between framework and application:

```
1. Framework pre-generated     ← base
2. Third-party packages        ← extend / override framework  
3. Application generated       ← extend / override everything
```

## Per-Layer Cache File Structure

```
# Framework cache — pre-generated, ships with framework
valkyrja/cache/
  AppContainerData.php        ← all framework container bindings
  AppEventData.php            ← all framework event listeners
  AppHttpRoutingData.php      ← all framework HTTP routes
  AppCliRoutingData.php       ← all framework CLI routes

# Third-party package cache — pre-generated, ships with package
some-valkyrja-package/cache/
  AppContainerData.php        ← package container bindings
  AppEventData.php            ← package event listeners
  AppHttpRoutingData.php      ← package HTTP routes (if any)
  AppCliRoutingData.php       ← package CLI routes (if any)

# Application cache — generated by valkyrja-build
app/cache/
  AppContainerData.php        ← all application container bindings
  AppEventData.php            ← all application event listeners
  AppHttpRoutingData.php      ← all application HTTP routes
  AppCliRoutingData.php       ← all application CLI routes
```

## Framework Source Shipping Policy

The build tool requires access to provider source files to extract bindings and
handlers. This means:

- **PHP / Python** — source always present on disk via Composer / pip. No
  special requirement.
- **Go** — Go module system downloads full source via `go mod download`. No
  special requirement.
- **Java** — Valkyrja publishes `-sources.jar` as a **required** build
  dependency, not optional. The build tool mandates it.
- **TypeScript** — Valkyrja ships `.ts` source files alongside compiled `.js`.
  Not just `.d.ts` declaration files — full source.

This is a framework policy decision. Source availability is a hard requirement
for full cache generation. Third-party packages built on Valkyrja must follow
the same policy.

## Provider Contracts

All five languages implement the same provider contract shape. The build tool
reads the return values of these methods directly from AST — no execution
required.

### ComponentProviderContract

The top-level aggregator. Returns lists of sub-providers by category.

**PHP** (reference implementation)

```php

{
    public static function getContainerProviders(ApplicationContract $app): array;
    public static function getEventProviders(ApplicationContract $app): array;
    public static function getCliProviders(ApplicationContract $app): array;
    public static function getHttpProviders(ApplicationContract $app): array;
}

// implementation
class HttpComponentProvider implements ComponentProviderContract
{
    public static function getContainerProviders(ApplicationContract $app): array
    {
        return [
            HttpContainerProvider::class,
            HttpMiddlewareProvider::class,
        ];
    }

    public static function getEventProviders(ApplicationContract $app): array
    {
        return [
            HttpEventProvider::class,
        ];
    }

    public static function getCliProviders(ApplicationContract $app): array
    {
        return [];
    }

    public static function getHttpProviders(ApplicationContract $app): array
    {
        return [
            HttpRouteProvider::class,
        ];
    }
}
```

Container bindings provider. `publishers()` returns a map of binding key to

only.

```php
    public static function getRoutes(): array;
```

## Route Parameters and Compiled Regex

Routes with dynamic segments require `Parameter` objects defining each segment's
name and pattern. The build tool extracts this data from AST, runs the
framework's own `ProcessorContract::route()` to compile the regex, and writes
the pre-compiled result into the aggregated `AppHttpRoutingData` class. See "The
Four Generated Data Classes" above for the full `HttpRoutingData` structure.

### Two Sources of Parameter Data

**Path 1: Parameter objects inline in the route definition**

Parameter objects appear as inline constructor calls — AST reads them as static
literals:

```php
// PHP
HttpRoute::get('/users/{id}/posts/{postId}', $handler, [
    new Parameter('id',     pattern: '[0-9]+'),
    new Parameter('postId', pattern: '[0-9]+'),
])
```

```java
// Java
HttpRoute.get("/users/{id}/posts/{postId}",handler, List.of(
        new Parameter("id",     "[0-9]+"),
    new

Parameter("postId","[0-9]+")
))
```

```go
// Go
data.Get("/users/{id}/posts/{postId}", handler,
parameter.New("id", "[0-9]+"),
parameter.New("postId", "[0-9]+"),
)
```

```python
# Python
HttpRoute.get('/users/{id}/posts/{postId}', handler, [
    Parameter('id', pattern='[0-9]+'),
    Parameter('postId', pattern='[0-9]+'),
])
```

```typescript
// TypeScript
HttpRoute.get('/users/{id}/posts/{postId}', handler, [
    new Parameter('id', '[0-9]+'),
    new Parameter('postId', '[0-9]+'),
])
```

The AST reads each `Parameter` constructor call as a node with literal string
arguments:

```
ObjectCreation: Parameter
  arg[0]: 'id'      ← string literal ✅
  arg[1]: '[0-9]+'  ← string literal ✅
```

**Path 2: Parameter annotations on controller action methods**

For annotated controllers (PHP, Java, Python) parameters live on the method
alongside `#[Handler]` / `@Handler` / `@handler`:

```php
// PHP
#[Handler(static fn(ContainerContract $c, array $args): Response
    => $c->getSingleton(UserController::class)->show($args[0], $args[1]))]
#[Parameter('id',     pattern: '[0-9]+')]
#[Parameter('postId', pattern: '[0-9]+')]
public function show(int $id, int $postId): Response {}
```

```java
// Java
@Handler((ContainerContract c, List < Object > args) ->
        c.

getSingleton(UserController .class).

show((int) args.

get(0), (int)args.

get(1)))

@Parameter(name = "id", pattern = "[0-9]+")
@Parameter(name = "postId", pattern = "[0-9]+")
public Response show(int id, int postId) {
}
```

```python
# Python
@handler(
    lambda c, args: c.get_singleton(UserControllerClass).show(args[0], args[1]))
@parameter('id', pattern='[0-9]+')
@parameter('postId', pattern='[0-9]+')
def show(self, id: int, post_id: int) -> Response:
    pass
```

Both paths produce identical data — a list of parameter name/pattern pairs
readable as literal AST nodes. ✅

### How the Build Tool Uses Parameter Data

The build tool does **not** construct the regex itself via AST logic. Instead:

1. AST extracts path string, parameter list, handler, method, and all route
   metadata
2. Build tool constructs a plain `ValkyrjaRoute` object from the extracted
   data — enough for the processor
3. Build tool passes the constructed route to `ProcessorContract::route()` —
   unchanged, same as runtime
4. Processor returns the route with compiled regex filled in
5. Build tool reads all pre-compiled data and writes the `HttpRoutingData`
   instance as source text

The build tool runs framework code (`ProcessorContract`) during cache
generation. This is valid — the build tool is compiled from the framework source
and runs in the same language environment.

---

### The Build Tool as a Text Generator

The build tool is a **source code generator** — it writes strings that are valid
source code. It does not need to instantiate, extend, or have access to
application-defined custom route classes at generation time. It simply writes
their names as text.

This is exactly how every pre-compilation step works in every language:

- **Java annotation processor** — generates source files referencing application
  classes that don't exist yet when the processor runs. The compiler resolves
  them in the final compile pass.
- **Go code generation** — `go generate` writes `.go` files importing packages.
  `go build` resolves them.
- **TypeScript pre-step** — generates `.ts` files importing application types.
  `tsc` resolves them.
- **PHP build tool** — writes `.php` files extending application classes. PHP
  resolves them via autoloader at runtime.

A developer writing `extends AuthenticatedRoute` in their editor doesn't need
`AuthenticatedRoute` compiled and available to type the name. The build tool has
the same freedom.

---

### Custom Route Child Classes

If a developer defines a custom route child class and returns it from
`getRoutes()`:

```php
// developer's custom route
class AuthenticatedRoute extends HttpRoute {}

// returned from getRoutes() — build tool reads this from AST
return [
    new AuthenticatedRoute('/users/{id}', $handler, [...]),
];
```

The AST sees `AuthenticatedRoute` as a string — the class name of the
constructor call. The build tool:

1. Constructs a plain `ValkyrjaRoute` from the extracted data for regex
   compilation via `ProcessorContract`
2. Writes `AuthenticatedRoute` by name into the generated `HttpRoutingData`
   source text — no instantiation needed

The build tool does NOT need `AuthenticatedRoute` compiled in. It just writes
the name. The compiler/runtime resolves it when the generated file is
compiled/loaded alongside the application.

```
✅ new AuthenticatedRoute('/path', $handler, [...])   — class name read as string from AST
✅ AuthenticatedRoute::forAdmin('/path', $handler)    — static factory, class name read from AST
❌ $routeClass::get('/path', $handler)               — variable class name, not readable
❌ $this->buildRoute('/path', $handler)              — dynamic construction, not readable
```

### The Generated AppHttpRoutingData Class

The build tool aggregates **all** HTTP routes from **every** `HttpRouteProvider`
across all components and the application into a single `AppHttpRoutingData`
class. This is the one class the framework loads for all HTTP routing. The route
objects are written as source text using actual class names from the AST — the
build tool does not need those classes compiled in at generation time.

**PHP**

```php
// generated — ALL http routes for the entire application, all indexes pre-built
// written as source text — custom route types referenced by name, resolved at runtime

return new \Valkyrja\Http\Routing\Data\HttpRoutingData(
    routes: [
        // routes from UserHttpRouteProvider (via UserController annotation)
        'user.show' => new \App\Http\Routing\AuthenticatedRoute(
            path:       '/users/{id}',
            method:     'GET',
            parameters: [
                new \Valkyrja\Http\Routing\Data\Parameter('id', '[0-9]+'),
            ],
            handler: static fn(\Valkyrja\Container\ContainerContract $c, array $args): \App\Http\Response
```

### Build Tool Contract for Parameters

Parameter definitions follow the same simple literal rule as all other build
tool-readable data:

```
✅ new Parameter('id', '[0-9]+')               — inline constructor with literals
✅ new Parameter('id', pattern: '[0-9]+')      — named argument with literal
✅ #[Parameter('id', pattern: '[0-9]+')]       — annotation with literals
✅ @Parameter(name = "id", pattern = "[0-9]+") — annotation with literals

❌ new Parameter($name, $pattern)              — variable references
❌ new Parameter('id', $this->getPattern())    — method call
❌ $params = getParams(); route(..., $params)  — variable reference to parameter list
```

---



The build tool enforces one rule across all languages for any method it reads:

```
method body must contain exactly:
  one return statement
  returning a single list / array / map literal containing only:
    - class references (::class / .class / ClassName / string constants)
    - object instantiations (new Route(...) / Route::get(...))
    - method references ([self::class, 'method'] / ClassName::method / p.Method)
    - no variables
    - no method calls other than constructors and static factories
    - no conditionals
    - no loops
```

Error on violation:

```
Error: UserServiceProvider::publishers() must return a simple map literal.
Conditional logic and dynamic computation are not supported in provider methods.
The build tool requires static analysis of this method's return value.
See: https://valkyrja.io/docs/providers#build-tool-compatibility
```

## Publishers AST Readability

The `publishers()` method returns a map of binding key to method reference. The
build tool reads the map then resolves each method reference and reads that
method's body:

```
publishers() map literal
        ↓
resolve each method reference to its source location
        ↓
PHP / Java: read #[Handler] / @Handler annotation on method → extract closure
Go / TypeScript / Python: read method body directly → extract function
        ↓
resolve all type references to FQN
        ↓
generate cache data class
```

This works because both the map and the referenced method bodies are simple,
static, readable via AST without execution.

## Without Cache — All Languages Work

Every language works without cache data files. This is guaranteed by the
provider contract design:

- **PHP, Java, Python** — class references (`::class`, `.class`, class objects)
  allow the framework to instantiate providers and call methods directly at
  runtime
- **TypeScript** — constructor reference type (`Array<new () => Contract>`)
  allows direct instantiation and method calls at runtime — no string lookup, no
  registry needed
- **Go** — interface methods called directly on provider structs at runtime — no
  reflection needed

```typescript
// TypeScript runtime — no cache, direct instantiation and method calls
for (const ComponentProvider of componentProviders) {
    const component = new ComponentProvider()
    for (const HttpProvider of component.getHttpProviders(app)) {
        const provider = new HttpProvider()
        for (const route of provider.getRoutes()) {
            router.register(route)  // ✅ works without cache
        }
    }
    for (const ServiceProvider of component.getContainerProviders(app)) {
        const provider = new ServiceProvider()
        for (const [key, publisher] of Object.entries(provider.publishers())) {
            container.bind(key, publisher)  // ✅ works without cache
        }
    }
}
```

This means the build tool and cache generation infrastructure can be built
incrementally — the framework ships and works correctly in all languages while
the build tool is developed. Cache is a performance optimization, not a
correctness requirement.

## PHP Cache CLI Command — Near-Term Breaking Change

PHP is the only language with a working cache generation mechanism today — the
`cache:generate` CLI command. However this command will break the moment
closure-based handler logic is implemented because:

- The current command relies on the old dispatch-based route registration
- Closure handlers cannot be serialized by the existing cache mechanism
- The `#[Handler]` annotation and AST extraction via nikic/php-parser are not
  yet implemented

**The PHP CLI command is a known TODO** — it needs to be migrated to the
valkyrja-build tool approach before handler logic ships. Until then PHP runs
without cache in the same way as all other languages, which is fully supported.

## Design Note — Why Routes and Listeners Cannot Use a Publisher-Style Map

When designing the `ServiceProviderContract`, a natural question arose: could
routes and listeners be expressed the same way as container bindings — a
`publishers()`-style map of identifier to handler method, with the build tool
reading method bodies directly from AST? This would eliminate `getRoutes()` /
`getListeners()` entirely and unify all provider patterns into a single
map-based approach.

This was considered and rejected for a fundamental architectural reason.

**Container bindings are simple key→factory pairs.** The binding key IS the
complete identity. The factory closure is the only additional data needed. A
complete cache entry can be generated from just the method body.

**Routes are multi-dimensional data structures.** A route carries HTTP method,
path pattern, dynamic segment constraints, regex compilation data, middleware
chain, name/alias, parameter defaults, host constraints, and scheme
constraints — all in addition to the handler. The
`HttpRoute::get('/users/{id}', handler)` call populates all of these fields
together as a complete object. Decomposing this into a key/method-body map loses
all the metadata the router needs to build its dispatcher trie, compile route
regexes, and resolve middleware chains.

**Listeners have the same problem.** A listener carries event type binding,
priority, and stop-propagation behavior alongside the handler. A flat key/body
map cannot express these.

This is why `getRoutes()` and `getListeners()` return complete object lists
while `publishers()` uses a map — the difference reflects a genuine
architectural distinction. Routes and listeners are data; container bindings are
mappings.

## Discussion Summary

The data cache file solution emerged from recognizing that at framework scale
the provider chain traversal has real and meaningful cost — not just routes but
container bindings for every component the application uses. The original
assumption that container bindings are cheap to register was revised when
considering a full application using Http, Container, Event, CLI, Queue, Cache,
Session, and Auth components each with their own service providers.

The provider tree discovery problem was solved by requiring provider list
methods to return simple list literals readable by AST without execution. This
is a hard framework convention — the build tool cannot work with dynamic or
conditional provider lists. The tradeoff is an explicit constraint on how
providers are written in exchange for build-time static analysis.

A separate `valkyrja.yaml` file was initially proposed as the build tool entry
point. This was superseded by recognizing that the application config class
already contains the authoritative list of component providers — the same class
the developer uses to create the application. The build tool reads the config
class via AST, eliminating a duplicate source of truth and a maintenance burden.
The component provider constants class was also dropped — it would have allowed
developers to pass constant references into the config which the build tool
cannot resolve statically without executing code. Binding key constants files
are unaffected.

The framework source shipping policy was established to solve the compiled
language problem: Java providers compiled into a jar have no readable source for
the build tool. The solution — requiring -sources.jar for Java and .ts source
for TypeScript — is a framework policy decision that makes source a first-class
artifact rather than an optional convenience.

The per-package cache file structure with load order override was established to
preserve the provider system's extensibility. Rather than one monolithic cache
file, each package generates its own cache files. The application always loads
last and can override any framework or package binding. This mirrors exactly how
the provider system works at runtime.

The publishers() AST readability question — whether the handler annotation
pattern works with the build tool — was resolved by recognizing that the build
tool reads the publishers map (simple literal, readable) then separately reads
each referenced method's annotation or body (also readable). The indirection
through a method reference is transparent to the AST walker.

The single-pass compile for all compiled languages was achieved by establishing
that the build tool runs before the compile step, not after. Combined with the
application config class provider tree walk, this eliminates the two-pass
compile that was considered early in the design — no bootstrap binary is needed
because the build tool can discover everything via static AST analysis of source
files.
